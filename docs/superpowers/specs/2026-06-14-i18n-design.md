# Internationalization (i18n) Design

> **Status:** Design approved, plan ready
> **Locale:** `en` (default), `nb` (Norwegian Bokmål), `pl` (Polish)
> **Library:** `next-intl` v3

## Architecture

**next-intl** v3 with App Router subpath routing. Locale prefix on all routes:
`/en/`, `/nb/`, `/pl/`. Middleware detects locale from cookie → `Accept-Language`
header → falls back to `en`.

**Message format:** ICU, stored as JSON in `frontend/messages/{locale}.json`.

**Scope:** Full app — all 13 components, metadata, theme names, slide labels
(~190 strings).

## Why next-intl

- Gold standard for Next.js 15 App Router i18n
- Server + client component support out of the box
- Subpath routing with zero-config middleware
- ICU message format with plural/gender support
- Lightweight (~40kB), tree-shakeable

## next-intl Setup

### `i18n/request.ts`

```ts
import { getRequestConfig } from "next-intl/server";

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`../../messages/${locale}.json`)).default,
}));
```

### `middleware.ts`

```ts
import createMiddleware from "next-intl/middleware";

export default createMiddleware({
  locales: ["en", "nb", "pl"],
  defaultLocale: "en",
  localeDetection: true,
});

export const config = {
  matcher: ["/((?!api|_next|_vercel|.*\\..*).*)"],
};
```

### `next.config.mjs` change

```js
import createNextIntlPlugin from "next-intl/plugin";
const withNextIntl = createNextIntlPlugin();
// ... existing config wrapped: export default withNextIntl(nextConfig);
```

### `[locale]/layout.tsx`

- Wraps children in `<NextIntlClientProvider messages={messages} locale={locale}>`
- Sets `<html lang={locale}>` dynamically
- Imports `getMessages`, `getLocale` from `next-intl/server`

## Component Integration Pattern

### Server Component (layout.tsx)

```tsx
import { NextIntlClientProvider } from "next-intl";
import { getMessages, getLocale } from "next-intl/server";

export default async function LocaleLayout({ children, params: { locale } }) {
  const messages = await getMessages();
  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

### Client Component

```tsx
import { useTranslations } from "next-intl";
const t = useTranslations("saveButton");
// ... {t("save")} ...
```

### Language Switcher

```tsx
import { useLocale } from "next-intl";
import { usePathname, useRouter } from "next-intl/client";

export function LocaleSwitcher() {
  const locale = useLocale();
  const pathname = usePathname();
  const router = useRouter();

  const switchTo = (nextLocale: string) => {
    router.replace(`/${nextLocale}${pathname}`);
  };

  return (
    <select value={locale} onChange={(e) => switchTo(e.target.value)}>
      <option value="en">EN</option>
      <option value="nb">NB</option>
      <option value="pl">PL</option>
    </select>
  );
}
```

## Edge Cases

- **Accept-Language unsupported locale** → middleware falls back to `en`
- **Theme names** (Ocean, Forest, Citrus) — brand proper names, kept as-is
- **Font pairing names** — label translated, font names kept as-is
- **Dynamic slide labels** via `getSlideLabel` — returns translation key, component renders via `t(getSlideLabel(type))`
- **Share links without locale prefix** — middleware catches `/s/abc` → redirects to `/en/s/abc`
- **`aria-label` attributes** — translated alongside visible text
